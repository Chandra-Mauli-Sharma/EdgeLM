# Week 1 — Wiring real llama.cpp inference

Phase 0 proved the shared-runtime memory thesis with a placeholder token loop.
This step replaces that loop with **real llama.cpp inference**, so the tokens are
actually generated by the model and the tokens/sec number becomes meaningful.

**The seam is clean:** only the native layer changes. `EdgeLMService.kt`,
`NativeBridge.kt`, the SDK, and both demo apps stay exactly as they are — they
already stream whatever the native side emits. You are swapping the engine under
a stable JNI boundary.

Copy-ready files live in `docs/week1/`:
`llama_runner.h`, `llama_runner.cpp`, `CMakeLists.txt`.

---

## Step 1 — Vendor llama.cpp as a submodule

From the project root (Git Bash, or PowerShell with git installed):

```bash
git submodule add https://github.com/ggerganov/llama.cpp \
    runtime-service/src/main/cpp/llama.cpp
cd runtime-service/src/main/cpp/llama.cpp
# PIN a known-good commit so the API can't drift under you:
git checkout b4200        # or any recent release tag; then:
cd ../../../../..
git submodule update --init --recursive
```

> **Why pin?** The llama.cpp API changes often. The provided `llama_runner.cpp`
> matches the `examples/simple/simple.cpp` shape from late-2024/2025 releases. If
> you check out latest and a symbol doesn't resolve, open
> `llama.cpp/examples/simple/simple.cpp` and copy the current call shape — that
> example is the canonical minimal reference and is kept in sync with the API.

## Step 2 — Swap in the three native files

Overwrite the Phase 0 placeholders with the real versions:

```powershell
copy docs\week1\llama_runner.h   runtime-service\src\main\cpp\llama_runner.h
copy docs\week1\llama_runner.cpp runtime-service\src\main\cpp\llama_runner.cpp
copy docs\week1\CMakeLists.txt   runtime-service\src\main\cpp\CMakeLists.txt
```

`edgelm_jni.cpp` does **not** change — it already calls `load_model` /
`generate` / `request_cancel` / `unload_model`, and those signatures are identical.

What changed inside `llama_runner.cpp`:

- `load_model` now calls `llama_model_load_from_file` (weights still mmap'd, so
  the Phase 0 shared-memory result holds — `use_mmap = true`).
- `generate` tokenizes a ChatML-wrapped prompt, runs a real
  `llama_decode` loop with a top-k/top-p/temp sampler, detokenizes each token
  with `llama_token_to_piece`, and streams it through the same `Sink`.
- Cancellation and EOG (end-of-generation) stops are handled.

## Step 3 — Build config notes

- `runtime-service/build.gradle.kts` already sets `abiFilters += "arm64-v8a"` and
  the CMake path — no change needed. Add `"x86_64"` only if you also want the
  emulator (llama.cpp builds fine for it, just slower).
- Keep `-DANDROID_STL=c++_static`; llama.cpp links cleanly into the single
  `libedgelm.so`.
- First build compiles all of ggml/llama — **expect several minutes**. Incremental
  builds are fast. The `.so` grows to a few MB; that's the engine, as intended.
- The `CMakeLists.txt` disables tests/examples/tools/server, OpenMP (no libgomp in
  the NDK), and llamafile — the combination that builds reliably under the NDK.

## Step 4 — Build, push, run

```powershell
.\gradlew.bat :runtime-service:installDebug :demo-app-a:installDebug :demo-app-b:installDebug
# model.gguf is already in place from Phase 0; if not, re-run:
# .\tools\push_model_and_install.ps1 "C:\path\to\qwen2.5-0.5b-instruct-q4_k_m.gguf"
```

Open Demo A, tap Generate. Instead of the repeating placeholder sentence you
should now see a **real, coherent answer** streaming token by token.

## Step 5 — Re-measure against the gate

```powershell
adb shell logcat -s edgelm-native   # watch model load + thread count
.\tools\measure_memory.ps1          # memory should look like Phase 0
```

Now the throughput number in the app is real. Interpreting it:

| Model (CPU, arm64) | Rough tokens/sec | Verdict |
|---|---|---|
| Qwen2.5-0.5B Q4 | ~40–100+ | Easily clears the >30 gate |
| Llama-3.2-1B Q4 | ~20–50 | Around the gate |
| Llama-3.2-3B Q4 | ~5–15 | Below 30 on CPU — **this is expected** |

The 3B on CPU sitting under 30 tok/s is **not a failure** — it's the argument for
Phase 1's GPU/NPU backends (Vulkan, then QNN). For the Phase 0/Week 1 gate, prove
>30 tok/s with a small model (0.5B/1B); note the 3B number as the CPU baseline the
accelerated backends must beat.

## Gotchas & tuning

- **API drift is the #1 issue.** If the build fails on a llama.cpp symbol, diff
  against `examples/simple/simple.cpp` at the commit you pinned. The header
  comment in `llama_runner.cpp` lists the recent renames.
- **Chat template.** `format_prompt()` hardcodes ChatML (correct for Qwen2.5). For
  other model families, switch to `llama_chat_apply_template()` with the model's
  built-in template, or the output will look off.
- **Threads.** We use ~half the cores (big cluster). Try `n_threads = big-core
  count` and measure; more isn't always faster once you hit the little cores.
- **Context per request.** For the spike we create/free a `llama_context` each
  call (stateless). Phase 1 pools contexts and keeps KV warm per session (this is
  where the paged-KV and prompt-cache work from the architecture doc lands).
- **First-token latency.** Prompt prefill dominates cold latency; it scales with
  prompt length. Keep the system prompt short while benchmarking decode speed.
- **Battery/thermal.** Sustained CPU decode heats the SoC; the real scheduler +
  governors (Part 8 of the architecture doc) are what make this safe at scale.

## What this unlocks next (Phase 1 shortlist)

1. **GPU backend** — build ggml with Vulkan; set `n_gpu_layers > 0`; re-measure.
2. **OpenAI HTTP shim** — a localhost server over the same `generate()` path.
3. **Context pooling + KV reuse** — turn the stateless per-call context into warm,
   per-session state (paged KV, prompt cache).
4. **Real scheduler** — replace the single-thread FIFO executor with priority +
   preemption at token boundaries.

See `docs/edgelm-architecture.html` for how each of these is designed.
